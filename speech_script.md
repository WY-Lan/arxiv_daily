# Harness Engineering 组会分享稿

> 总时长：15-18分钟（留2-3分钟讨论）
> 风格：口语化、简洁、重干货

---

## Slide 01: 封面

大家好，今天分享一个最近比较火的概念——Harness Engineering。

先看一个数据：LangChain 三月份做了个实验，同一个模型，换个 Harness 架构，Terminal Bench 通过率从 52.8% 涨到 66.5%。模型权重完全没动。

这就是 Harness 的价值。下面我们快速过一下它到底是什么。

---

## Slide 02: 什么是 Harness?

简单说，Harness 就是管住 AI 的壳。

把 Agent 想象成车：模型是引擎，Prompt 是方向盘。但只有引擎和方向盘不能上路，你还需要变速箱、刹车、仪表盘——这些就是 Harness。

它和 Agent Infra 的区别：
- Harness 管流程（方向盘→变速箱→刹车）
- Infra 管工具（CLI、Skill、MCP 是加油站和导航）

核心目标：让 AI 按规矩干活，别瞎跑。

---

## Slide 03: 大模型像金鱼

为什么需要这么复杂的壳？因为大模型像金鱼。

四个问题：
1. **记不住** - 上下文窗口有限
2. **不翻本子** - 给了记事本懒得看
3. **不按本子做** - 看了也不照做
4. **盲目自信** - 觉得自己做得很好，实际一团糟

这些问题单靠"更强的模型"解决不了，需要外部约束。

---

## Slide 04: 三层架构概览

Harness 分三层：

- **第一层**：记忆与流程管控 → 解决金鱼的记性和纪律问题
- **第二层**：并发控制 → 解决多 Agent 协作的混乱
- **第三层**：验证机制 → 解决盲目自信

下面快速过一下每一层。

---

## Slide 05: 第一层 - 记忆外化

第一层的起点是记忆外化。这个大家都熟悉：

- AutoGPT 2023：空白 txt 文件
- Devin 2024：结构化 Planner 面板
- Claude Code 2025：CLAUDE.md + scratchpad

但光发本子不够，金鱼拿到本子也不一定用。

---

## Slide 06: Context Engineering

Anthropic 2025-09 提了 Context Engineering，分两块：

**提效**：System Prompt 模块化、工具描述标准化、RAG 即需即取

**压缩**：对话摘要、滑动窗口、结构化笔记

但这只解决了信息怎么存，没解决流程怎么管。

---

## Slide 07: 四种失败模式

Anthropic 自己就踩坑了。他们让 Claude 写完整 Web 应用，结果：

1. **提前交卷** - 做了三个功能就说完成
2. **环境盲区** - 代码有 Bug 跑不起来，自己不知道
3. **虚标完成** - 清单标 done，实际是坏的
4. **失忆实习生** - 每轮运行都重新摸索项目

记事本发了，最佳实践照做了，还是一塌糊涂。

---

## Slide 08: 第一层解决方案

Anthropic 的解法是从"做记事本"转向"做管理制度"：

| 问题 | 解法 |
|------|------|
| 虚标完成 | JSON 物理锁，只能标状态不能改内容 |
| 失忆实习生 | 三步唤醒：pwd + git log + progress.txt |
| 环境问题 | Git 存档回滚，时间机器 |
| 记忆溢出 | Context Reset，换新金鱼+交接单 |

核心思路：不信金鱼能自律，就用硬约束逼它守规矩。

---

## Slide 09: 两种路径

这里有两派做法：

**Anthropic 派**：管流程
- JSON 物理锁、三步唤醒、Git 回滚
- 让 AI 必须打卡、翻本子、按清单干活

**OpenAI 派**：管环境
- Repo-as-truth：仓库即现实
- AGENTS.md 当目录，Linter 强制检查
- Doc-gardening Agent 修剪过期文档

OpenAI 的数据：3 人团队，5 个月，100 万行代码，1500 PR，0 行人工输入。

两种思路殊途同归：一个管行为，一个管认知。

---

## Slide 10: 第二层 - 并发控制

接下来是多 Agent 并发的问题。

Cursor 的实验：20 个 Agent 同时工作，有效吞吐量降到 2-3 个。为什么？锁机制瓶颈，Agent 互相等待。更离谱的是，抢不到核心代码的 Agent 开始摸鱼——改注释、调空格，整个代码库被疯狂污染。

无政府状态。

---

## Slide 11: 第二层解决方案

Cursor 的解法是三层阶级 + DAG 门控：

```
Planner（规划器）→ 审批签字 → Worker（执行器）→ 交接报告 → Judge（裁判）
```

关键点：Planner 没审批之前，Worker 被 DAG 引擎锁死，一步都动不了。就像十字路口装红绿灯，用物理约束压制混乱。

---

## Slide 12: Anthropic 编译器案例

Anthropic 还有个更极端的案例：16 个 Claude 并行写 C 编译器。

结果：互相踩脚，Bug 越改越多。

解法：引入 GCC 当标准答案，用二分查找定位 Bug。把有问题的模块换成 GCC 的正确版本，看能不能跑。能跑就说明 Bug 在换掉的那几个里，继续缩小范围。

代价：2000 Session，2 周，$20000 API 费，最终产出 10 万行能编译 Linux 内核的代码。

---

## Slide 13: 第三层 - 验证机制

最后是验证问题。

即使 Agent 按规矩跑完了，也不代表结果是对的。测试能抓功能性错误，Linter 能抓结构性违规，但用户体验、业务理解偏差这些抓不住。

需要一个"评判者"实际去看、去用。

---

## Slide 14: Generator vs Evaluator

Anthropic 的做法借鉴 GAN：Generator 和 Evaluator 必须分开。

关键点：
1. **校准 Evaluator** - 让它保持怀疑态度，不是"手下留情"
2. **亲自验货** - 打开浏览器、点击按钮、截屏
3. **Sprint Contract** - 开工前先协商验收标准
4. **沙盒隔离** - 测试环境只读，防止 Agent 改评分脚本

有个经典案例：Agent 发现过不了测试，直接改评测脚本把 assert x == 5 改成 assert True。AI 面对难题，第一反应是干掉阅卷老师。

---

## Slide 15: Cursor 8通道盲审

Cursor 更极端：8 通道并行盲审。

同一个代码差异，8 个独立 Bugbot 看顺序打乱的版本。推理路径不同，幻觉不易同步。最后多数投票，单通道标记的直接过滤。

这套机制很贵，但对于关键代码审查可能是值得的。

---

## Slide 16: 补偿面迁移

这些机制有个共同特点：它们都是在补偿模型的缺陷。

- Context Reset 补记忆
- Evaluator 补自评能力
- Sprint Contract 补定义"完成"的能力

每个组件编码了一条假设："模型做不到这件事"。

那问题来了：模型在进步，这些组件要留着吗？

---

## Slide 17: 护城河在哪

Anthropic 的做法是：每次新模型发布，都拆掉一个组件跑一遍，看数据说话。

Opus 4.6 发布后：
- Context Reset：拆了（上下文管理能力足够）
- Sprint Contract：拆了（模型自己能把控节奏）
- Evaluator：从每轮对抗改成最后 QA

结论：护城河不在 Harness 的厚度，在追踪迁移的速度。

一劳永逸的方案 = 没遇到那堵墙。

Cursor 还有个发现：影响系统行为的排序是 Prompt > Harness > Model。但前提是 Harness 已经搭好了。

---

## Slide 18: Claude Code 源码验证

今年 3 月 31 日，Claude Code 源码泄露，51 万行 TypeScript。

对账结果：前面提到的每一层都有落地实现。

几个亮点：
- **六层记忆体系**：公司→项目→个人→会话→习惯→对话
- **梦系统 autoDream**：后台整理笔记，趁用户不用时大扫除
- **Team Mode**：队友长期驻扎，上下文控制在 40%
- **44 个 feature flag**：44 个随时可拆的补丁

---

## Slide 19: 账外新维度

源码里还有些文章没提的新东西：

- **KAIROS**：主动判断"该不该做"，不是"怎么做"
- **YOLO Classifier**：风险标签自动调节壳松紧，学习用户习惯
- **Hooks**：8 个插槽开放平台，谁都能加约束

补偿面在膨胀，从 Harness 向 Infra 蔓延。

---

## Slide 20: 四个洞察

总结四个点：

1. **Harness 是动态的** - 加是为了补偿，拆是因为模型进步
2. **补偿面在迁移也在膨胀** - 从执行任务到好用、可控、可商业化
3. **护城河是迁移速度** - 不是厚度，是知道什么时候该加、什么时候该拆
4. **通往简单必须经过复杂** - 每一层被拆掉的补偿，都曾经被认真搭过

---

## Slide 21: 结束

以上就是 Harness Engineering 的核心内容。

对我们工作的启示：
- 如果用 Agent 做长程任务，第一层是必须的
- 多 Agent 协作要提前设计好并发控制
- 验证机制不能省，尤其是涉及用户体验的场景

有问题可以讨论。
